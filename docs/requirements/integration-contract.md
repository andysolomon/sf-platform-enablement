# Integration Contract ‚Äî Salesforce ‚Üî FRS Service (Vercel mock)

> **Phase 2.3 deliverable.** The concrete API contract both sides build against: the Apex
> `CalloutService` (outbound) and inbound handler, and the Vercel mock service. It is the
> artifact that lets the two sides be developed *independently* and tested against each other.
>
> **Exercises:** integration deck `identify-functional-and-non-functional-integration-
> requirements`, `references.html` (Named Credentials, OAuth flows, idempotency).
> **Traces:** FR-5/FR-6/FR-7 (outbound), FR-8/FR-9 (inbound), FR-10 (correlation),
> NFR-1 (auth/secrets), NFR-2 (data minimization), NFR-3 (idempotency), NFR-5 (resilience).
>
> **Free-tier note:** the "FRS Service" is the Vercel mock; this contract is what the mock
> implements. It is versioned (`/v1`) and follows additive-only change rules (NFR-8 analogue
> for the wire contract ‚Äî breaking changes go to `/v2`).

---

## 1. Conventions (both directions)

| Convention | Rule |
|---|---|
| Transport | HTTPS only, JSON (`application/json; charset=utf-8`) |
| Versioning | Path-prefixed `/v1/...`; additive changes only within v1; breaking ‚Üí `/v2` |
| **Correlation id** | UUID generated by the *originator* of a business event and **echoed unchanged** across the whole round-trip (outbound ‚Üí FRS ‚Üí inbound). Header `X-Correlation-Id` + body `correlationId`. Enables FR-10 end-to-end trace. |
| **Idempotency key** | Unique per logical message (not per retry). Header `Idempotency-Key` + body `idempotencyKey`. The **receiver** dedupes on it (NFR-3). Retries of the *same* message reuse the *same* key. |
| Timestamps | ISO 8601 UTC, e.g. `2026-06-16T22:41:00Z` |
| Time budget | Outbound callout timeout **10s** (SF side). FRS must respond within it or SF treats it as a retryable timeout. |
| Auth | Per direction, ¬ß2. No secret ever in a payload or log (NFR-1, NFR-2). |

## 2. Authentication (per the landscape ¬ß6 auth matrix)

| Direction | Mechanism | Flow | Notes |
|---|---|---|---|
| **Outbound** SF ‚Üí FRS | Named Credential + External Credential | OAuth 2.0 **Client Credentials** (fallback: API key header) | SF Apex uses `callout:FRS_Service/...`; the credential injects `Authorization`. No URL/secret in code. |
| **Inbound** FRS ‚Üí SF | Connected App + least-privilege **integration user** | OAuth 2.0 **JWT Bearer** ‚Üí access token | FRS presents `Authorization: Bearer <token>`; SF authorizes as the integration user, scoped to only the inbound resource + target write. |

Auth failures (`401`/`403`) are **non-retryable** and raise an alert condition (NFR-1, FR-7).

## 3. Outbound ‚Äî `POST callout:FRS_Service/v1/notifications`

*S1, Remote Process Invocation ‚Äî Fire-and-Forget. SF notifies FRS of a business event.
SF does not depend on a business response; FRS acknowledges receipt only.*

### 3.1 Request (SF ‚Üí FRS)

Headers: `X-Correlation-Id`, `Idempotency-Key`, `Content-Type`, `Authorization` (via Named Credential).

```json
{
  "correlationId": "9f1c8b2e-3a44-4d77-9c10-7e2b5a0d11ef",
  "idempotencyKey": "todo-04t...-completed-1",
  "eventType": "todo.completed",
  "occurredAt": "2026-06-16T22:41:00Z",
  "source": "salesforce/platform-enablement",
  "payload": {
    "externalRef": null,
    "recordId": "a01gK00000XyZ12QAF",
    "status": "Done"
  }
}
```

### 3.2 Response (FRS ‚Üí SF)

**Success ‚Äî `202 Accepted`** (receipt acknowledged, async on FRS side):

```json
{ "receiptId": "rcpt_8c2...", "status": "accepted", "correlationId": "9f1c8b2e-..." }
```

`200 OK` is also accepted (treated as success).

### 3.3 Outbound status-code ‚Üí SF behavior

| FRS returns | SF interpretation | Retry? (NFR-5) | Logged as |
|---|---|---|---|
| `200` / `202` | Success | ‚Äî | success (FR-5) |
| `409 Conflict` (dup idempotency key) | Already received ‚Üí **treat as success** | No | deduplicated success |
| `400 Bad Request` | Malformed payload ‚Äî our bug | **No** | error, non-retryable, alert |
| `401` / `403` | Auth failure | **No** | error, non-retryable, **alert** (NFR-1) |
| `429 Too Many Requests` | Rate-limited | **Yes**, honor `Retry-After` | retrying |
| `5xx` | FRS transient | **Yes**, backoff | retrying |
| Timeout / connection error | Unknown | **Yes**, backoff | retrying |

Retries: max 3 (NFR-5 default), exponential backoff. On exhaustion ‚Üí dead-letter + replay
path (FR-7). Because the message carries a stable `idempotencyKey`, a retry that *actually
succeeded on the FRS side but whose response was lost* is safely de-duplicated by FRS.

## 4. Inbound ‚Äî `POST /services/apexrest/frs/v1/status`

*S2, Remote Call-In. FRS reports status back into Salesforce, which updates the related
record idempotently.*

### 4.1 Request (FRS ‚Üí SF)

Headers: `X-Correlation-Id` (echo of the originating outbound id, when applicable),
`Idempotency-Key`, `Content-Type`, `Authorization: Bearer <token>`.

```json
{
  "correlationId": "9f1c8b2e-3a44-4d77-9c10-7e2b5a0d11ef",
  "idempotencyKey": "frs-status-7741",
  "eventType": "status.updated",
  "occurredAt": "2026-06-16T22:41:05Z",
  "payload": {
    "externalRef": "rcpt_8c2...",
    "salesforceRecordId": "a01gK00000XyZ12QAF",
    "status": "Acknowledged",
    "detail": "Processed by FRS"
  }
}
```

### 4.2 Response (SF ‚Üí FRS)

**Applied ‚Äî `200 OK`:**
```json
{ "status": "applied", "recordId": "a01gK00000XyZ12QAF", "correlationId": "9f1c8b2e-..." }
```

**Duplicate (idempotent no-op) ‚Äî `200 OK`** (NFR-3, FR-9):
```json
{ "status": "duplicate", "recordId": "a01gK00000XyZ12QAF", "correlationId": "9f1c8b2e-..." }
```

### 4.3 Inbound status codes (SF ‚Üí FRS)

| SF returns | Meaning | FRS expected behavior |
|---|---|---|
| `200 applied` | Update applied | Done |
| `200 duplicate` | Already applied (same key) | Treat as success; do not resend |
| `400` | Validation error (missing/invalid fields) | Fix payload; non-retryable |
| `401` / `403` | Auth/authorization failure | Refresh token / check scope; non-retryable |
| `404` | `salesforceRecordId` not found | Non-retryable; investigate |
| `5xx` | SF transient | Retry with backoff |

SF persists `idempotencyKey` on first apply; a second call with the same key returns
`200 duplicate` with **no** second record mutation and a "deduplicated" log (FR-9 / NFR-3).

## 5. Field dictionary & data classification (NFR-2)

Every field that crosses the boundary, with its classification from `system-landscape.md` ¬ß5.
**Nothing above Internal crosses.**

| Field | Direction | Type | Classification | Notes |
|---|---|---|---|---|
| `correlationId` | both | UUID | Internal | trace key, non-sensitive |
| `idempotencyKey` | both | string | Internal | dedupe key |
| `eventType` | both | enum | Internal | `todo.completed`, `status.updated` |
| `occurredAt` | both | ISO-8601 | Internal | |
| `source` | outbound | string | Internal | originating system |
| `payload.recordId` / `salesforceRecordId` | both | SF Id | Internal | the related record |
| `payload.status` | both | enum | Internal | business status |
| `payload.externalRef` | both | string | Internal | FRS-side reference |
| `payload.detail` | inbound | string | Internal | short human note; **no PII, no secrets** |
| `Authorization` header | both | token | **Restricted** | never logged (NFR-1) |

Validation: a payload containing any field not in this dictionary is rejected `400`
(prevents accidental over-sharing). Logs persist everything **except** `Authorization` and
truncate `detail` to N chars.

## 6. Idempotency & correlation worked example (the FR-10 round-trip)

1. User completes a todo ‚Üí consumer publishes domain event with a **new** `correlationId`
   `C` and `idempotencyKey` `K1`.
2. Outbound `POST /v1/notifications` (key `K1`, corr `C`) ‚Üí FRS `202`. SF logs success (corr `C`).
3. FRS processes, then calls back inbound `POST /frs/v1/status` echoing corr `C`, new key `K2`.
4. SF applies the update, persists `K2`, returns `200 applied`. SF logs (corr `C`).
5. Network glitch ‚Üí FRS resends step 3 with the **same** `K2`. SF returns `200 duplicate`,
   no second update. SF logs deduplicated (corr `C`).
6. Operator queries logs by corr `C` ‚Üí full ordered chain visible (FR-10).

## 7. Contract versioning & change rules

- Additive within `/v1`: new **optional** fields, new enum values consumers can ignore.
- Breaking (removed/renamed field, new required field, changed type, changed auth) ‚Üí `/v2`,
  with `/v1` kept per the deprecation policy until consumers migrate (mirrors NFR-8 for the
  wire contract).
- Both the Apex contract test (Phase 7) and the Vercel mock assert against this document;
  drift in either fails the contract test.
